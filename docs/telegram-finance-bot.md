# FinTrack Telegram Bot

## Роль бота

Telegram bot є швидким мобільним інтерфейсом до фінансової системи. Він не дублює бізнес-логіку сайту: розпізнавання повідомлень, чернетки, збереження операцій і звіти винесені у серверний пакет `@resource-manager/server`.

## Меню

- Додати витрату
- Додати дохід
- Витрати за період
- Доходи за період
- Звіт по категоріях
- Баланс
- Останні операції
- Місячний підсумок
- Налаштування
- Допомога

Меню доступне як reply keyboard для швидкого мобільного доступу і як inline keyboard для сценаріїв з вибором.

## UX-сценарії

### Витрата

1. Користувач пише: `АТБ продукти 560`.
2. Сервер розпізнає тип `EXPENSE`, суму, дату, опис і категорію.
3. Бот показує чернетку з кнопками:
   - Підтвердити
   - Скасувати
   - Змінити суму
   - Змінити категорію
   - Змінити дату
4. Після підтвердження запис створюється через `createManualExpense`.

Якщо категорія неочевидна, бот показує варіанти категорій. Без категорії витрата не зберігається.

### Дохід

1. Користувач пише: `Зарплата 50000`.
2. Сервер розпізнає тип `INCOME`, суму, джерело, дату й опис.
3. Бот показує чернетку.
4. Після підтвердження запис створюється через `createIncome`.

### Звіти

Бот підтримує вільні запити:

- `Покажи витрати за сьогодні`
- `Скільки я витратив у квітні?`
- `Покажи витрати по категоріях за цей місяць`
- `Які були доходи за березень?`
- `Дай виписку з 1 по 15 квітня`
- `Покажи останні 10 витрат`
- `Скільки я витратив на їжу цього місяця?`
- `Порівняй березень і квітень`

## API

Сайт уже має основні mutation endpoints:

- `POST /api/expenses`
- `PATCH /api/expenses/:id`
- `POST /api/incomes`
- `PATCH /api/incomes/:id`
- `GET /api/dashboard/snapshot`

Додані read endpoints для звітів:

- `GET /api/reports/expenses?from=&to=&category=`
- `GET /api/reports/incomes?from=&to=`
- `GET /api/reports/categories?from=&to=`
- `GET /api/reports/balance?from=&to=`
- `GET /api/transactions/latest?limit=`
- `GET /api/summary/monthly?from=&to=`

Бот працює в API процесі й викликає серверні сервіси напряму. Якщо бот буде винесений в окремий контейнер, ці ж сервіси можна замінити HTTP-викликами до endpoint-ів без зміни UX-логіки.

## NLP

Серверний парсер виділяє:

- тип операції: витрата або дохід;
- суму;
- валюту, зараз за замовчуванням UAH `980`;
- дату: сьогодні, вчора, позавчора, `DD/MM/YYYY`, `DD.MM.YYYY`;
- категорію або джерело доходу;
- опис;
- період для звіту.

Поточна реалізація комбінує правила, категорії з бази та словник фінансових підказок. Наступний крок: додати AI fallback, який повертає строго валідований JSON, якщо правила не дали впевненого результату.

## Voice

Голосовий сценарій:

1. Бот отримує Telegram voice message.
2. Через Telegram Bot API `getFile` отримує тимчасовий шлях до файлу.
3. Файл завантажується в пам'ять, не зберігається на диск.
4. Telegram OGG/Opus конвертується в MP3 через `ffmpeg` у API контейнері.
5. Аудіо передається в OpenAI transcription.
6. Отриманий текст проходить через той самий NLP-flow.
7. Користувач підтверджує чернетку перед збереженням.

Рекомендований STT для старту: OpenAI `gpt-4o-mini-transcribe`. Причини: проста інтеграція з уже наявним OpenAI ключем, добра підтримка української, не треба окремо заводити Google/Azure credential flow. Для enterprise-режиму можна додати Google Speech-to-Text або Azure Speech Services як провайдери з тим самим інтерфейсом `transcribeAudio`.

## Безпека

- Telegram має бути прив'язаний до конкретного `User`.
- Секрети зберігаються через наявний encrypted secret механізм.
- Voice файли не пишуться на диск.
- Усі фінансові запити фільтруються по `userId`.
- Чернетки мають TTL 15 хвилин.
- Секрети через Telegram можна вводити тільки якщо явно увімкнено `ENABLE_TELEGRAM_SECRET_SETUP=true`.
- Для webhook потрібно використовувати `X-Telegram-Bot-Api-Secret-Token`.

## Етапи розвитку

1. Базовий UX: меню, витрати, доходи, підтвердження, звіти.
2. Голос через OpenAI transcription.
3. AI fallback для складних повідомлень зі strict JSON schema.
4. Налаштування персональних скорочень і синонімів категорій.
5. Більш багаті звіти: бюджети, цілі, попередження, аномалії.
6. Винесення бота в окремий контейнер, якщо навантаження або деплой цього потребуватиме.
